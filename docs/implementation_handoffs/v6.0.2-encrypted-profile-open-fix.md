# v6.0.2 Encrypted Profile Open Fix Handoff

## Summary
- Investigated GitHub issue 22, where an encrypted profile was created successfully but the application appeared to crash or terminate while opening it.
- Reproduced encrypted profile creation/open locally and confirmed the SQLCipher profile itself remained valid.
- Released the correction as v6.0.2 so v6.0.1 installations receive the fixed profile-open sequence through the normal updater.

## Root Cause
- Empty new profiles schedule the current-owner bootstrap prompt immediately after opening.
- The bootstrap loop retried until a party was selected, so dismissing the prompt on a new encrypted profile could trap the application during post-open finalization.
- The catalog-number registry selector also refreshed suggestions/status during catalog activation and allowed transient registry lookup exceptions to bubble into the profile load path.

## Changes
- `_ensure_owner_party_bootstrap` now performs one prompt per scheduled attempt and returns on cancel or empty selection, leaving the profile open.
- `CodeIdentifierSelector` now tolerates transient registry lookup/classification failures during refresh and status updates.
- Help troubleshooting notes now state that the current-owner prompt can be dismissed and the owner can be set later in Party Manager.
- Added regression coverage for owner-bootstrap cancel behavior, app-shell profile-open behavior, and code-registry widget refresh failure handling.
