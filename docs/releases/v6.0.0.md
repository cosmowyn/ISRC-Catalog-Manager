# ISRC Catalog Manager 6.0.0

[![Release](https://img.shields.io/github/v/tag/cosmowyn/ISRC-Catalog-Manager?label=release)](https://github.com/cosmowyn/ISRC-Catalog-Manager/releases/tag/v6.0.0)

Version: 6.0.0
Date: 2026-06-03
Type of update: Major

## Highlights
- Moved profile databases to a SQLCipher-backed encrypted architecture for new profiles and encrypted profile opens.
- Added password prompting, session-only password handling, optional safe OS keychain/keyring remember support, and an in-app change-password workflow for encrypted profiles.
- Added plaintext-profile migration handling: existing unencrypted SQLite profiles warn before opening, can be encrypted in place, can be opened unencrypted for the session, or can be cancelled.
- Ensured pre-encryption recovery backups are written to the app backup directory with backup metadata sidecars, so Storage Admin and recovery workflows classify them as retained recovery points.
- Added SQLCipher dependency coverage and migration regression tests proving encrypted databases migrate from legacy schema versions 1 through 44 to the current schema target 45.
- Updated Storage Admin and Help so manual database backups and profile-maintenance recovery backups are warning-protected rather than treated as routine reclaimable storage.

## Database Encryption
- New profile creation now asks for a database password and stores the password only in the current application session unless the user enables safe remembered-password storage.
- Existing encrypted profiles prompt for a password before opening. Wrong passwords are rejected before the profile is activated.
- Remembered database passwords use the operating-system keychain/keyring only when the backend is available and considered safe. Passwords are not written to normal settings, JSON files, the profile database, environment variables, or source files.
- The Application Settings dialog includes database-password controls and a change-password action for SQLCipher profiles.
- The database connection factory now opens SQLCipher connections whenever a session password is available for the profile path, while still supporting explicitly allowed plaintext profile opens.

## Migration and Backup Safety
- Plaintext profile migration creates a verified encrypted copy, validates the result, and replaces the active database only after verification succeeds.
- The previous plaintext database is retained as a recovery backup under the app `backups` directory, using a `.db` backup file plus `.db.backup.json` sidecar metadata.
- Backup metadata records the source profile and profile-maintenance origin so Application Storage Admin can classify the backup as a retained recovery point.
- If sidecar metadata cannot be written after successful encryption, the encrypted migration remains successful and the user receives a warning instead of a false encryption failure.
- Manual database backups are no longer recommended for automatic cleanup. They require explicit warning-approved deletion. Restore safety copies remain eligible for cleanup after the restore workflow is no longer needed.

## Schema Compatibility
- SQLCipher-backed schema initialization reaches the current schema target.
- Encrypted legacy-start regression coverage now runs `migrate_schema()` from every `PRAGMA user_version` value from 1 through 44 and verifies schema target 45 plus SQLite integrity.
- Existing schema migration QA continues to cover detailed legacy data-shape migrations in the standard SQLite test suite.

## Reporting and Build Support
- Added `sqlcipher3==0.6.2` and `keyring==25.7.0` to runtime dependency metadata and bootstrap requirements.
- Updated package smoke and build metadata so SQLCipher and keyring dependencies are expected in packaged builds.
- Continued report-proxy hardening for large diagnostic payloads and packaged reporting configuration.

## Compatibility Notes
- This is a major architectural release. Encrypted profiles require a database password to open.
- Users should retain migration recovery backups until they have verified the encrypted profile and their normal backup posture.
- Python remains pinned to the 3.14.4 runtime/test target.
