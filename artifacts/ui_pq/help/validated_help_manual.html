Overview

A clear product overview of what the app manages, why it exists, and how the main workflows fit together.

Music Catalog Manager is a local-first desktop workspace for building and maintaining a serious music catalog. It brings together recording metadata, releases, musical works, contracts, rights, parties, documents, deliverables, GS1 product data, diagnostics, history, and quality control in one application.

It is designed for independent artists, labels, managers, and catalog owners who need more than a basic track list and want a reliable system for both metadata and catalog operations.

The app is organized around a few core ideas:

  • Profiles: each profile is a self-contained catalog database, so separate catalogs remain clean and portable.
  • Add Track: the primary single-track creation entry point. Every new track must either link to an existing Work or create a new Work from the track before save.
  • Add Album: the primary batch-entry surface. It behaves as batch Add Track, so each populated row resolves Work governance before the album save completes.
  • Work Manager: the parent governance and management surface for work metadata, ownership, contributions, and linked tracks after or alongside track-first entry.
  • Releases: first-class product records for UPC/EAN, release artwork, ordering, and release-level metadata.
  • Code Registry Workspace: the central workspace for internal catalog, contract, and license numbers plus the separate Registry SHA-256 Key and external identifiers.
  • Works: a composition layer that stays distinct from recordings so the same work can connect to multiple tracks.
  • Parties: reusable people and companies for writers, publishers, labels, managers, licensees, and organizations.
  • Contracts and rights: lifecycle-aware agreement records, obligations, document versions, and explicit rights positions linked back to the catalog.
  • Assets: managed deliverables and artwork variants with approval and primary-version tracking.
  • Catalog Table: the central browser for searching, selecting, bulk editing, and reviewing recording data.
  • Global Search: a relationship-aware search surface across works, tracks, releases, contracts, rights, parties, documents, and assets.
  • Docked catalog workspace: code registry, promo-code ledger, release, work, license, party, contract, rights, deliverables, and search panels can stay open as tabbed workspace surfaces beside the catalog table.
  • Import, exchange, and conversion: CSV, XLSX, JSON, XML, ZIP, audio-tag, Bandcamp promo-code, template conversion, bulk audio attach, and GS1 workflows for bringing data in, classifying internal versus external identifiers, reconciling it, filling external templates, attaching media, exporting it, and archiving it safely.
  • Metadata Standards: GS1, ISRC, ISWC, UPC/EAN, catalog identifiers, audio tags, exchange schemas, and template conversion are kept together conceptually so standards-driven metadata work stays reviewable.
  • SoundCloud publishing: private-by-default SoundCloud upload, update, and existing-upload linking workflows are prepared from catalog metadata, reviewed in a manual preflight, and keep OAuth secrets out of SQLite and normal settings.
  • Quality Dashboard: a practical readiness view for metadata gaps, identifier conflicts, broken media links, rights risks, and operational blockers.
  • Diagnostics and recovery: snapshots, backups, cleanup, trim, diagnostics, repair paths, and logs keep heavier workflows recoverable.
  • Action Ribbon: a customizable quick-action strip for your most-used commands.
  • Background tasks: longer scans, imports, exports, snapshots, and file operations run outside the UI thread to keep the workspace responsive.
  • Settings and history: identity, registration settings, themes, undo/redo, snapshots, diagnostics, and logs.
  • Startup feedback: a Qt-native splash reports milestone-based startup progress while storage reconciliation and workspace restore complete.
  • Flexible file storage: file-backed records can be kept as database BLOBs or as managed local files without changing the surrounding UI workflow.
  • Media badge icons: separate visual indicators for stored audio and image BLOBs can be configured with system icons, emoji, or compressed custom images.

The menu bar mirrors those workflows. File handles profiles, exchange, and template conversion, Edit starts governed musical entry with Add Track and Add Album and edits the current selection, Catalog opens the richer repertoire tools, Settings controls app and profile configuration, View manages layout and helper surfaces, History protects recoverability, and Help gives you diagnostics, logs, and this manual.

Main Window

How the main workspace is laid out, how the docks behave, and how to move through the app efficiently.

The main window is designed as a practical catalog workspace rather than a single fixed screen. The Catalog Table stays central, Add Track and Add Album are the primary governed entry points for new musical items, and Work Manager stays available as the governance and follow-up surface once those entries exist.

  • Action ribbon: a customizable strip of high-frequency actions. Use View > Customize Action Ribbon… to make it match your workflow.
  • Profiles toolbar: switch databases, create a new profile, browse to an external profile, reload the profile list, or remove the selected entry.
  • Profiles ribbon toggle: use View > Show Profiles Ribbon when you want that toolbar visible or hidden, and the choice is remembered with the rest of the workspace.
  • Dockable panes: keep the window focused on your current task by showing only the panes you need.
  • Tabbed catalog tools: Code Registry Workspace, Promo Code Ledger, Release Browser, Work Manager, Party Manager, Contract Manager, Rights Matrix, Deliverables and Asset Versions, and Global Search open as docked tabs beside the table so you can keep using the catalog inventory while those panels remain open.
  • Publishing entry points: Catalog > Publish > SoundCloud… and catalog-table context menus open the SoundCloud preflight workflow without bypassing manual review.
  • Diagnostics cleanup tools: stored artist and album cleanup lives with Diagnostics so integrity review and cleanup follow-up stay together.
  • Saved layout: column layout, dock placement, and visibility preferences are remembered so the app opens the way you work.

Use the menus when you need the full surface of the product, or stay inside the docked views for everyday entry and review. The window title, branding, and appearance can be customized from Settings > Application Settings. When you open multiple catalog tools, the app prefers tabbed docking so the workspace stays compact and easier to navigate while the table remains usable.

Workflow Playbooks

Start-to-finish examples for workflows that depend on each other, written for users who do not already know catalog, rights, accounting, or publishing terminology.

Workflow Playbooks UI: Current qualified UI reference for the Workflow Playbooks chapter. This image is regenerated by the Help documentation qualification workflow.

This chapter shows complete chains rather than isolated button descriptions. A chain matters because many catalog tasks depend on earlier records: a royalty statement needs a party, a track, a work, a contract, usage lines, a posted calculation, and then a payout; a SoundCloud upload should start from a governed track with media and should leave a traceable publication record.

Playbook 1: Create a First Single From Nothing

Before you start: decide which profile should hold the catalog. A profile is the database file for one catalog, label, artist, or project. Have the audio file, artwork, release date, artist name, and any known identifiers ready. If you do not know the ISRC, ISWC, UPC, or catalog number yet, leave them blank or use the internal registry tools later.

  1. Use the profile controls at the top of the main window to create or open the correct profile.
  2. Open Add Track from the action ribbon, Edit menu, or Catalog workspace menu.
  3. Enter the track title, main artist, genre, album or single title, release date, and track number.
  4. Attach the audio file and artwork if they are available. Choose database storage when portability matters more than database size, or managed-file storage when large media should stay outside the SQLite file.
  5. Resolve Work governance before save. Choose Create New Work From Track when this is a new composition, or Link to Existing Work when the recording belongs to an existing composition.
  6. Save the track. The app creates the track row and the governed Work link in the same user workflow.
  7. Open the Catalog Table and search for the title. Confirm the row exists and that media indicators appear where audio or artwork was attached.

Expected result: the catalog contains a track, any linked media is stored in the selected storage mode, and the track is not orphaned because it has a Work decision. If a release title was supplied, release-aware workflows can build on that metadata.

Troubleshooting: if Save is blocked, check required fields and Work governance. If generated ISRC preview is disabled, configure the ISRC prefix in Application Settings. If media cannot be attached, verify the file still exists and is a supported audio or image format.

Related workflows: Add Album, Work Manager, Release Browser, Catalog Media Attach, Audio Tags, Code Registry Workspace, and Data Quality Dashboard.

Playbook 2: Build an Album With Release and Work Links

Before you start: gather the album title, UPC/EAN if assigned, release date, artwork, ordered track list, artist names, and audio files. Decide whether each track is a new composition or a recording of an existing Work.

  1. Open Add Album. Enter the shared album information once in the overview section.
  2. Add enough track tabs for the complete album. Each populated tab represents one track row.
  3. For each track tab, enter title, artist, track number, duration, optional identifiers, and optional audio file.
  4. For each populated track, choose the Work outcome. Use new Work creation for new compositions and existing Work linking for alternate versions, remixes, or re-recordings.
  5. Review duplicate track-number warnings. They warn because unusual releases exist, but you should still correct accidental duplicates.
  6. Save the album. The app creates the release/product record and links the saved tracks to it.
  7. Open Release Browser to verify the release, ordered track list, UPC/EAN, artwork, and release metadata.

Expected result: one release record exists with ordered tracks, and every saved track has its own Work governance outcome. Shared album art is stored once and linked where appropriate.

Troubleshooting: if a row was skipped, confirm it was not blank. If a track is missing from the release, open Release Browser and add selected tracks. If imported identifiers do not match the internal format, store them as external identifiers rather than forcing them into the internal registry.

Related workflows: Release Browser, Work Manager, Code Registry Workspace, GS1 Metadata, Export Selected Exchange, and Quality Dashboard.

Playbook 3: Import, Review, Clean, and Export Catalog Data

Before you start: identify whether your source is a catalog exchange file, a third-party sheet, an audio folder, or embedded audio tags. These are different workflows. Exchange import writes catalog rows; template conversion fills an external template; audio tags read metadata from files; catalog media attach connects existing rows to media.

  1. Use File > Import & Exchange for CSV, XLSX, JSON, XML, or ZIP package import into the catalog.
  2. Preview rows, choose the matching strategy, map columns, and decide whether to dry run, create, update, merge, or insert only new rows.
  3. Run a dry run first when the source is new or high risk. Review skipped fields, duplicate matches, identifier classifications, and custom-field decisions.
  4. Run the real import only after the preview explains what will happen.
  5. Open Data Quality Dashboard and fix missing titles, duplicate identifiers, broken media paths, missing artwork, invalid barcodes, orphaned rights, and other blockers.
  6. Use Export Selected Exchange or Export Full Exchange when the catalog data should leave the app in the app's exchange schema.
  7. Use File > Conversion when the target is a rigid third-party template. Load the template, map catalog fields, preview output, and export a new filled file.

Playbook 6: Verify a Degraded Audio File Honestly

Before you start: understand the lifecycle. A file may start as a watermarked WAV, be uploaded to SoundCloud, be transcoded internally for streaming, be ripped by another downloader as MP3, then be converted back to WAV for analysis. That chain can lower confidence and sync score, but it does not mean the watermark is absent.

  1. Open Verify Audio Authenticity or Inspect Forensic Watermark.
  2. Select the suspicious file directly. If the file is not WAV, the app converts it to a temporary WAV analysis copy.
  3. Read the conversion notice. The notice should say lossy source material, stream rips, and transcoding can lower confidence and sync scores.
  4. Interpret the result bands. Verified means confidence is at least 85%. Likely match means confidence is at least 70% and sync score is at least 65%. Low-confidence candidate means confidence is at least 55%. A likely degraded lossy derivative can still be meaningful evidence.
  5. Use the key id, matched database record, confidence, sync score, group agreement, repeat groups, and conversion notes to decide the next operational step.

Expected result: the report states what was found, what the score means, and what may have reduced certainty. A confidence above 50% is treated as an indication of a database watermark candidate rather than as proof that nothing was found.

Troubleshooting: if no candidate appears, verify that the open profile contains the relevant watermark/export ledger. If confidence is low, try to inspect the least-transcoded file you have. If the source is a SoundCloud rip, document that lifecycle in the case notes.

Related workflows: Audio Authenticity, SoundCloud Publishing, Media Conversion, Derivative Ledger, and Application Log.

Profiles and Databases

How catalog profiles work, how to switch between them, and why they make the app safe and portable.

Profiles and Databases UI: Current qualified UI reference for the Profiles and Databases chapter. This image is regenerated by the Help documentation qualification workflow.

A profile is one self-contained catalog database. New profiles are protected with SQLCipher encryption, which means the profile file and encrypted backups cannot be read as ordinary SQLite files if they are copied from the machine.

  • Profile selector: choose the active database from the profile combo box in the toolbar. Changing the selection changes which catalog is open.
  • New…: create a fresh local encrypted profile database. The password must be at least 12 characters and is kept only for the current app session unless remembered login is enabled.
  • Browse…: open an existing database file from any location. Encrypted profiles prompt for the database password before the profile opens.
  • Reload List: refresh the known profile list from disk.
  • Remove…: open a removal dialog, choose the profile to delete from the dropdown, then confirm the final delete prompt before that database file is removed from disk.

When you browse to an older unencrypted SQLite profile, the app warns before opening it and offers a migration path. You can encrypt the profile now, open it unencrypted for the session, or cancel. If you choose to keep one known unencrypted profile unencrypted, the dialog checkbox suppresses future warnings only when you open that profile. Application Settings also includes a separate global switch to turn off all unencrypted-profile warnings at your own caution. Migration creates a SQLCipher copy, preserves the original schema version for the encrypted copy, verifies the encrypted result, and replaces the active database only after the encrypted copy is valid. Invalid legacy ISRC values that cannot satisfy current validation are cleared during migration and noted in the affected track comments instead of blocking encryption. The previous unencrypted file is retained as a recovery backup in the app backups folder so the operation can fail safely.

Profile-specific catalog data stays with the profile. Shared app-level conveniences such as saved themes and remembered layout settings stay available across profiles. When switching to an encrypted profile, the password prompt unlocks and prepares the target database before the active workspace is retargeted; cancelling the password prompt leaves the current profile open. If the last-used startup profile cannot be prepared or opened safely, the app leaves that database untouched, opens a fallback or startup recovery profile, and shows the original path and failure reason so you can browse, restore, or report the damaged profile from a running workspace. SQLCipher secure-memory locking is an optional deployment hardening and is not enabled during normal profile opening, because some Windows environments reject the required memory-lock request during profile switching; this does not change database-at-rest encryption. On first launch, the app can also offer to open Application Settings so you can configure registration values, database-password posture, snapshot posture, and appearance before deeper catalog work begins.

For a non-technical user, the safest mental model is: a profile is the file that contains the catalog. Opening another profile is like opening another project. Before importing, issuing invoices, exporting media, or publishing to SoundCloud, verify that the correct profile name is visible. If you are unsure, use About, Diagnostics, or the profile path shown in support surfaces to confirm the database location before making changes.

Core Concepts

  • Royalty statement: an immutable statement artifact generated from a posted royalty calculation.

  1. Open Party Manager and make sure the customer, owner, payee, VAT/tax, billing address, and payout details exist where required.
  2. Open Template Workspace, select an invoice template, choose the buyer/customer party, select invoice catalog rows for each indexed invoice line, and fill quantities or use the travel calculator to apply kilometres to a travel row.

  5. Record payments as money arrives. The invoice status changes from unpaid to partially paid to paid from linked payment rows.
  6. If a correction is needed, create a credit note against the issued invoice instead of editing financial history silently.
  7. Create a royalty calculation from usage lines, review the detail, approve and post it, then generate a royalty statement.
  8. Record artist payouts until the payable balance is settled. Use reports to confirm balances and command logs.

Audio Authenticity

Export direct master authenticity copies, export provenance-only derivatives, manage signing keys, and verify authenticity honestly inside the catalog.

Audio Authenticity UI: Current qualified UI reference for the Audio Authenticity chapter. This image is regenerated by the Help documentation qualification workflow.

The audio authenticity workflow combines two layers: a compact keyed watermark embedded into exported audio plus an Ed25519-signed manifest that carries the real authenticity claim. The watermark links the file back to the catalog record; the signature proves which manifest was issued.

  • Settings > Audio Authenticity Keys…: generate or review local signing keys. Public keys are stored in the profile, while private keys stay as local key files under the app settings root.
  • Catalog > Export Authenticity Watermarked Audio…: write WAV, FLAC, or AIFF master copies that keep the original canonical source unchanged, embed a compact keyed watermark token, write the catalog metadata tags that are already available for the track, and save a sibling .authenticity.json sidecar with the signed direct-authenticity manifest.
  • Catalog > Export Authenticity Provenance Audio…: copy supported lossy derivatives as-is, write the available catalog tags, and save a signed provenance sidecar that binds the exported derivative back to a previously verified watermarked master.
  • Catalog > Audio > Authenticity & Provenance > Verify Audio Authenticity…: inspect either selected catalog audio or a chosen external file. Direct WAV/FLAC/AIFF inputs are checked as-is; other decodable audio formats are converted to a temporary WAV analysis copy before watermark verification. Signed provenance lineage sidecars are still verified directly for supported derivatives.
  • Deliverables and Asset Versions > Derivative Ledger: review managed export batches after the export moment, filter by format/kind/status, inspect derivatives, details, and lineage, and launch authenticity verification directly from the selected derivative.

This feature is not DRM and does not promise forensic certainty. A watermark always changes the waveform slightly, so the goal is perceptual transparency rather than mathematical identity. The strongest in-app verification happens when the open profile still contains the original reference audio, because the app can compare the inspected export against that stored source directly.

Direct embedded authenticity exports remain WAV, FLAC, and AIFF master copies, but verification can inspect decodable lossy or streamed derivatives by converting them to a temporary WAV analysis copy. When conversion is used, the report notes that confidence and sync scores can be affected by lossy encoding while still allowing watermark discovery. Verification reports classify direct watermark evidence as verified, likely match, low-confidence candidate, or not found; confidence above 50% is reported as positive database watermark evidence rather than a clean negative.

The separate Registry SHA-256 Key in the Code Registry Workspace is not part of this authenticity system. It does not replace watermark keys, authenticity signing keys, or the signed provenance workflow described here.

Recipient-specific forensic watermarking for leak tracing is a separate managed export workflow. It remains distinct from authenticity: authenticity proves signed linkage to a canonical master record, while forensic delivery exports focus on tracing shared copies. Forensic exports support WAV plus managed lossy delivery formats, include a SoundCloud-specific export path, and report verified, likely, low-confidence, and not-found evidence bands instead of treating every degraded score as a clean negative.

Connection and secret handling

  • Settings shortcut: SoundCloud Settings… opens Application Settings directly on the SoundCloud tab.
  • Safe settings: normal settings may store the client id and redirect URI. Client secret, access token, refresh token, auth code, authorization headers, and full OAuth callback URLs are never shown back to the user, stored in SQLite, or written to normal settings.
  • OS keychain/keyring: when a safe backend is available, SoundCloud credentials and token bundles are stored in the operating system keychain/keyring.
  • Session-only fallback: if no safe keychain/keyring is available, the app uses in-memory session-only storage and clearly reports that reconnect is required after restart.
  • Disconnect: disconnect clears local SoundCloud token and client-secret state and marks the account disconnected. It does not delete catalog tracks, publication history, or existing SoundCloud uploads.

Preflight review

  • Readiness table: the dialog shows track title, operation, metadata readiness, audio/artwork readiness, warnings, blocking errors, planned sharing state, and known remote publication status.
  • Blocking errors: disconnected accounts, missing titles, missing primary audio, explicit quota exhaustion, oversized audio, and invalid media conditions block publishing until fixed.
  • Warnings: quota uncertainty, rate-limit state, release-date conflicts, metadata conflicts, unsupported optional fields, omitted internal notes, or unclear update eligibility are shown without hiding the plan.
  • Safe defaults: sharing defaults to private, downloadable remains false, and streamable remains true.

Metadata sent to SoundCloud

  • Core fields: title and the prepared audio file are required. Supported optional fields include genre, ISRC, release date, artwork, label name, release title, album title, description, tag list, purchase URL, artist, publisher, composer, barcode/UPC/EAN, ISWC, P line, contains music, explicit-content status, and supported per-run permission flags.
  • Editable review: description, tags, buy link, label, release, explicit-content, contains-music, comment, reveal, and related fields can be reviewed or edited in the publish/update flow where the UI exposes them.
  • Unsupported fields: the app avoids sending fields that are not accepted by the supported SoundCloud request schema and warns when a requested value cannot be written through the current API path.
  • Private catalog notes: internal comments, contracts, rights notes, QA notes, private notes, credentials, and callback details are never published.

Audio, artwork, and upload execution

  • Embedded or managed media: SoundCloud publishing can resolve audio and artwork from database-stored BLOBs or managed local files, matching the same catalog media model used by preview and export workflows.

Background Tasks

How the app keeps long operations responsive and safe without risking the catalog database.

Background Tasks UI: Current qualified UI reference for the Background Tasks chapter. This image is regenerated by the Help documentation qualification workflow.

Long-running operations run outside the UI thread so the workspace stays responsive while real work is happening.

  • Central task runner: imports, exports, ZIP packaging, snapshots, restores, quality scans, catalog-audio copy export, backup, and integrity checks are dispatched through one shared Qt background-task manager.
  • Main-thread UI updates only: worker threads report back through Qt signals, and dialogs, tables, messages, and status text are updated on the main thread only.
  • Per-thread SQLite connections: background jobs never reuse the main window's SQLite connection. Each worker opens and closes its own connection safely.
  • SQLite concurrency: the app enables foreign-key enforcement and a busy timeout. Unencrypted profiles use WAL mode, while encrypted SQLCipher profiles use rollback-journal mode to avoid fragile encrypted WAL sidecars during profile switching and restart. Write-heavy jobs are serialized per profile so concurrent background writers do not fight each other.
  • Progress and cancellation: longer jobs show a progress dialog or in-place status text where practical. Some file-based jobs, such as catalog-audio copy export, can be cancelled safely.
  • Safe shutdown: the app blocks closing while background jobs are still running so restores, imports, and other file/database writes cannot be interrupted mid-operation.

If a background task fails, the app reports it clearly, logs the details, and rolls database work back cleanly. The result is a system that feels responsive without becoming fragile.

Automatic Reporting

Automatic reporting means a packaged app build can detect that the previous session ended without a clean shutdown and offer to prepare a crash report the next time it starts. Source-tree and unfrozen development runs skip this startup crash-report check so normal code restarts do not create repeated prompts. The app does not send reports silently. In packaged builds, startup asks whether you want to review the crash report, then opens the same privacy-safe report workflow used by Help > Report a Bug….

  • Crash detection: an unclean shutdown can seed a crash report with app version, platform, recent safe log context, and the detected failure state.
  • Optional OS context: the crash prompt and manual bug-report form include an unchecked option for bounded operating-system log context. When you enable it, the app uses read-only native log queries with no shell execution, no privilege elevation, short timeouts, small output limits, and local sanitisation before preview or submission.
  • Manual reports: Report a Bug… collects your summary, description, reproduction steps, expected result, and actual result.
  • Preview first: both paths build a structured GitHub issue, sanitise the payload locally, show the exact Markdown preview, and let you copy it before submission.
  • Submit Report: submission happens only after you press this button. If a configured report proxy accepts the payload, it can create the GitHub issue for you without requiring a local GitHub login.
  • Local fallback: if no secure proxy is configured, the app is offline, the proxy is rate-limited, or submission fails, the sanitised report is saved under the app data reports folder for later review.

Installed release builds can include a public HTTPS report proxy endpoint through the bundled resources/reporting.json file. That file is generated from release configuration and must not contain credentials; it only tells the app where the server-side reporting endpoint lives. Environment configuration can override the bundled file for development or support testing. The app never ships a bug-report token, shared account password, GitHub private key, or other repository write credential. Large diagnostic reports are shortened before online GitHub submission so the issue body stays within GitHub limits, and repeated real proxy submission failures can trigger a short cooldown. Missing local proxy configuration does not consume that cooldown once the proxy has been configured.

The sanitiser redacts tokens, passwords, API keys, connection strings, email addresses, phone numbers, local user names, and private home-folder paths. Reports must not be used to send catalog databases, raw royalty or contract data, private documents, or audio files. If a workflow depends on sensitive catalog records, describe the steps and visible symptoms rather than pasting private source data.

Do not edit logs as a way to fix the catalog. Logs are evidence. If a log points to a database, managed-file, or settings problem, use the relevant workflow to repair the source issue and then keep the log as the explanation of what happened.
